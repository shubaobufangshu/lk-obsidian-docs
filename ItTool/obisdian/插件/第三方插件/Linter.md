Obsidian Linter 是目前 Obsidian 生态中最强大、最常用的 Markdown 格式规范化插件。它可以自动统一你的笔记格式，包括 YAML 元数据、标题层级、列表缩进、链接格式、代码块、段落间距等，让你的整个知识库保持风格一致，极大提升可读性和可维护性。
## 核心配置详解
### 基础全局设置
这是 Linter 最核心的控制开关，决定了格式化的触发方式和基础行为。

| 配置项 | 功能说明 | 推荐设置 | 注意事项 |
|--------|----------|----------|----------|
| **保存时格式化文件** | 按下 `Ctrl+S`/`Cmd+S` 保存文件时自动运行格式化；Vim 模式下使用 `:w` 命令也会触发 | ✅ 开启 | 这是最推荐的使用方式，养成"保存即格式化"的习惯 |
| **文件修改时格式化** | 关闭当前文件或切换到其他文件时，自动格式化刚刚编辑的文件 | ❌ 关闭 | 可能会在你不知情的情况下修改文件，导致意外的格式变化 |
| **无修改时不显示消息** | 如果文件已经符合所有格式规则，格式化后不显示"已格式化"的提示弹窗 | ✅ 开启 | 减少不必要的界面干扰 |
| **覆盖默认地区语言** | 设置 Linter 使用的语言，影响日期格式、标点符号等本地化处理 | 默认 (zh-cn) | 中文用户不要修改，否则可能出现日期格式错误 |
| **YAML aliases 样式** | 控制 YAML 元数据中 `aliases`（别名）字段的数组格式 | 单行数组 | 简洁美观，适合大多数情况；如果别名很长很多，可以改为多行数组 |
| **YAML tags 样式** | 控制 YAML 元数据中 `tags`（标签）字段的数组格式 | 单行数组 | 建议与 aliases 保持一致，统一风格 |
| **默认转义字符** | 当 YAML 值中包含特殊字符需要转义时，默认使用单引号还是双引号 | 单引号 | 单引号内的内容不会被 YAML 解析器额外转义，更安全可靠 |
| **当使用 YAML 多行数组时删除不必要的转义字符** | YAML 多行数组和单行数组的转义规则不同，开启后自动删除多余的转义字符 | ✅ 开启 | 避免不必要的转义导致 YAML 格式混乱 |
| **Latex 块标记符 $ 数量** | 设置将内容识别为块级 LaTeX（而非行内 LaTeX）所需的 `$` 符号数量 | 2 | 符合标准 Markdown 语法：`$$...$$` 表示块级公式，`$...$` 表示行内公式 |
### 忽略规则配置
这是**极其重要**的配置，用于排除不需要格式化的文件和文件夹，避免破坏特殊文件的格式。

1. **忽略文件夹**
   - 功能：指定不需要格式化的文件夹路径，每行输入一个
   - 必须添加的路径：
```
.obsidian
.trash
```
   - 建议添加的路径：
 ```
 附件
 模板
 归档
 node_modules
 ```
   - 注意：路径是相对于库根目录的，不需要加开头的 `/`

2. **忽略文件**
   - 功能：指定不需要格式化的单个文件，支持**正则表达式**
   - 常用示例：
 ```
 README.md
 首页.md
 \d{4}-\d{2}-\d{2}\.md # 忽略所有日期格式的日记文件
 */模板/*.md # 忽略所有模板文件夹下的md文件
 ```
   - ⚠️ 重要警告：如果你不熟悉正则表达式，请**不要使用**，否则可能会误排除大量文件

3. **额外文件扩展名**
   - 功能：除了 `.md` 之外，还需要 Linter 检查格式化的文件扩展名
   - 常用示例：`mdx`, `svx`
   - 注意：只有 Obsidian 本身视为 Markdown 的文件才会被真正格式化，其他扩展名只会进行语法检查
### 其他关键配置项
除了上面的基础配置，Linter 还有数十项针对 Markdown 不同元素的精细配置，下面是最常用、最值得调整的部分：
#### YAML 元数据配置
这是 Linter 最强大的功能之一，可以帮你自动管理和统一所有笔记的元数据。

1. **强制 YAML 开头**
   - 功能：确保所有文件都以 YAML 元数据块开头
   - 推荐：✅ 开启（如果你使用 YAML 元数据管理笔记）

2. **YAML 键排序**
   - 功能：自动对 YAML 元数据中的键进行排序
   - 推荐：✅ 开启，并设置自定义排序顺序：
 ```
 title
 aliases
 tags
 date
 updated
 author
 description
 ```
   - 好处：所有笔记的元数据顺序完全一致，一目了然

3. **自动添加日期字段**
   - 功能：自动在 YAML 中添加 `date`（创建日期）和 `updated`（更新日期）字段
   - 推荐：✅ 全部开启，并设置日期格式为 `YYYY-MM-DD HH:mm:ss`
   - 特性：`updated` 字段会在每次保存文件时自动更新

4. **移除空的 YAML 键**
   - 功能：自动删除 YAML 中值为空的键
   - 推荐：✅ 开启（保持元数据整洁）
#### 标题格式配置
1. **标题前后空行**
   - 功能：确保标题前后都有一个空行（文件开头的第一个标题除外）
   - 推荐：✅ 开启（大幅提升可读性）

2. **标题层级递增**
   - 功能：禁止标题层级跳跃（例如不能从 H1 直接跳到 H3）
   - 推荐：✅ 开启（规范笔记结构，便于大纲视图和目录生成）

3. **标题末尾标点**
   - 功能：自动移除标题末尾的句号、感叹号、问号等标点
   - 推荐：✅ 开启（标题通常不需要末尾标点）
#### 列表格式配置
1. **列表缩进大小**
   - 功能：统一列表的缩进空格数
   - 推荐：**2 个空格**（Markdown 标准，兼容性最好）
   - 注意：**绝对不要使用 Tab 缩进**，会导致严重的兼容性问题

2. **有序列表自动编号**
   - 功能：自动重新编号有序列表
   - 推荐：✅ 开启（避免手动编号错误）

3. **无序列表符号统一**
   - 功能：将所有无序列表符号统一为同一种
   - 推荐：**-**（减号），这是最常用、最美观的无序列表符号

#### 链接与图片配置
1. **内部链接格式**
   - 功能：统一 Obsidian 内部链接的格式
   - 推荐：**最短路径**（`[[文件名]]` 而非 `[[文件夹/文件名]]`）
   - 注意：如果有重名文件，Linter 会自动使用完整路径

2. **移除链接中的空格**
   - 功能：自动将链接中的空格替换为 `%20`
   - 推荐：✅ 开启（提升链接在不同平台的兼容性）

3. **图片大小统一**
   - 功能：自动为所有图片添加统一的宽度或高度
   - 推荐：可以设置为 `|400`（所有图片宽度为 400px），避免图片过大影响阅读
#### 代码块配置
1. **强制代码块语言标识**
   - 功能：要求所有代码块都添加语言标识（例如 ```python）
   - 推荐：✅ 开启（启用语法高亮，提升代码可读性）

2. **代码块前后空行**
   - 功能：确保代码块前后都有一个空行
   - 推荐：✅ 开启（提升可读性）
#### 段落与换行配置
1. **段落之间空行**
   - 功能：确保段落之间只有一个空行
   - 推荐：✅ 开启（避免过多空行浪费空间）

2. **移除行尾空格**
   - 功能：自动删除行尾的多余空格
   - 推荐：✅ 开启（Markdown 中只有两个连续空格表示换行，多余的空格没有意义）

3. **文件末尾空行**
   - 功能：确保文件末尾有且只有一个空行
   - 推荐：✅ 开启（符合大多数文本编辑器和 Git 的规范）
## 日常使用方法

### 1. 手动格式化单个文件
- **快捷键方式**：按下 `Ctrl+Alt+L`（Windows/Linux）或 `Cmd+Option+L`（Mac）
- **命令面板方式**：打开命令面板（`Ctrl+P`/`Cmd+P`），搜索 **Linter: Lint the current file**

### 2. 保存时自动格式化
- 开启"保存时格式化文件"开关后，每次保存文件都会自动运行格式化
- 这是最推荐的使用方式，无需额外操作

### 3. 批量格式化所有文件
- 打开命令面板，搜索 **Linter: Lint all files in the vault**
- ⚠️ 重要警告：这会格式化你整个库中的所有文件（除了忽略列表中的）
- 建议：**先备份你的库**，或者先在一个小文件夹中测试配置效果

### 4. 临时禁用格式化
- **禁用单个文件**：在文件的 YAML 元数据中添加：
  ```yaml
  linter: false
  ```
- **禁用文件中的某段内容**：用特殊注释包裹：
  ```markdown
  <!-- linter-disable -->
  这段内容不会被 Linter 格式化
  <!-- linter-enable -->
  ```
